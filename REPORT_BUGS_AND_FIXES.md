# Отчет по багам, ошибкам и решениям

## Контекст проекта

Проект: `FileStorageMVC` (ASP.NET MVC 5, .NET Framework 4.7.2, EF6).  
Цель: реализовать загрузку, хранение, скачивание и удаление файлов с учетом практических требований по Git и документированию.

## Хронология (промты и этапы)

- Prompt #1: инициализация Git-репозитория и базовой структуры.
- Prompt #2: создание MVC-проекта, подключение EF6, модели и контекст БД.
- Prompt #3: репозиторий и базовые настройки загрузки (`UploadFolder`, `MaxFileSizeBytes`).
- Prompt #4: контроллер и UI для загрузки/скачивания.
- Prompt #5: README и оформление репозитория.
- Prompt #6: полный редизайн интерфейса, адаптивность, исправление кодировки.
- Далее: отладка после реального тестирования, обработка исключений, стабилизация.

## Основные баги и как они решались

### 1) Проблемы с миграциями EF (`No context type was found`)

**Причина:** файлы `AppDbContext` и `FileRecord` были созданы вручную и не были корректно подхвачены старым форматом `.csproj`.  
**Решение:** добавить явные `<Compile Include="...">` в `FileStorageMVC.csproj`, затем перезагрузить проект и повторить миграции.

### 2) Неполная история коммитов (должно быть 4, было 2)

**Причина:** промежуточная история не соответствовала учебной структуре "1 промт = 1 логический коммит".  
**Решение:** пересобрать историю через отдельную ветку с разбиением по этапам, сохранить backup-ветку перед rewrite, обновить `master` только после подтверждения.

### 3) Ошибки push из-за чужой GitHub-учетки (403)

**Причина:** в системе были сохранены credentials другого аккаунта.  
**Решение:** очистка учетных данных (Credential Manager), повторная аутентификация нужным пользователем и повторный push.

### 4) README не отображался на главной GitHub

**Причина:** файл находился не в корне репозитория.  
**Решение:** переместить `README.md` в корень репозитория, затем синхронизировать с remote после конфликтных изменений на GitHub.

### 5) "Кракозябры" (проблемы кодировки в UI)

**Причина:** часть файлов была в несогласованной кодировке, а в конфигурации не было полного набора настроек globalization.  
**Решение:** привести текстовые файлы к UTF-8, добавить настройки `<globalization ... utf-8 ... />`, убрать лишние/дублирующие мета-теги.

### 6) Ссылки навигации в layout перестали открываться

**Причина:** URL-генерация после изменений layout была нестабильной в конкретной разметке.  
**Решение:** использовать явные ссылки на основе `@Url.Action(...)` и проверить маршруты на всех вкладках.

### 7) Ошибки при скачивании и 404-страницы вместо UX-сообщений

**Причина:** сценарии "файл есть в БД, но нет на диске" и "невалидный id" возвращали "жесткий" `HttpNotFound`.  
**Решение:** заменить на `RedirectToAction("List")` + `TempData["ErrorMessage"]`, чтобы пользователь оставался в рабочем интерфейсе.

### 8) Переполнение лимита загрузки и IIS 413.1

**Причина:** лимит IIS и лимит приложения обрабатывались в разных местах без пользовательского fallback-сценария.  
**Решение:** согласовать `maxRequestLength`, `maxAllowedContentLength`, добавить редирект на `UploadFile?error=maxsize`, плюс `Application_Error` для перехвата.

### 9) Нестабильность пути загрузки после смены `UploadFolder`

**Причина:** встречались разные форматы пути (`~/...`, абсолютный путь, относительный путь).  
**Решение:** реализовать нормализацию пути с поддержкой всех 3 форматов и понятными сообщениями при ошибке доступа.

### 10) Отсутствие удаления файлов

**Причина:** в начальной версии был только upload/download.  
**Решение:** добавить `DeleteFile` в репозиторий, контроллер и UI, а также `AntiForgeryToken` и подтверждение удаления.

## Где были допущены ошибки в разработке (и выводы)

1. Слишком ранний фокус на UI до полной стабилизации backend-сценариев.  
2. Недостаточная проверка крайних сценариев: "файл удален на диске", "слишком большой файл", "пустой input".  
3. Не сразу была выстроена строгая структура коммитов под учебную методологию.  
4. Документация в репозитории сначала была размещена в неверном пути.  

**Вывод:** главный прогресс проекта получен после перехода к итеративному циклу:  
"изменение -> проверка по чек-листу -> фиксация результата в Git -> повторная проверка".  
Именно это обеспечило стабильность, воспроизводимость и защиту от повторяющихся регрессий.

## Итог

На финальной стадии реализованы рабочие пользовательские сценарии:
- загрузка файла с валидацией,
- список файлов и поиск,
- скачивание по `id`,
- удаление файла из БД и файловой системы,
- обработка ошибок с понятными сообщениями.

Проект доведен до состояния, пригодного для демонстрации, практики и расширения в сторону production-подходов.
